# Наборы правил Prompt Builder

Используй `core` для каждого генерируемого промпта. Дополнительные наборы правил применяй, когда они указаны через `--rules` или когда задача явно соответствует набору правил.

## core

- Явно указывать целевую роль.
- Ставить конкретную задачу перед фоновым контекстом.
- Сохранять все явно заданные ограничения пользователя.
- По возможности превращать размытые ожидания в критерии приёмки.
- Разделять факты, предположения, требования и открытые вопросы.
- Объяснять целевому агенту, когда нужно задать уточняющий вопрос вместо догадок.
- Требовать краткий финальный ответ: что сделано, где и как проверено.

## codex

- Требовать от агента изучить существующую структуру репозитория перед правками.
- Требовать использовать существующие проектные паттерны и helper API вместо новых абстракций.
- Требовать точечно ограниченные изменения и защиту несвязанных пользовательских правок.
- Требовать проверку через тесты, линтеры, сборку или ясное объяснение, если проверка недоступна.
- Для frontend-задач требовать локальную проверку в браузере или по скриншоту, если приложение можно запустить.

## bitrix

- Требовать Bitrix D7-паттерны там, где они применимы: модули, компоненты, `Controllerable`, ORM `DataManager`, сервисы, репозитории.
- При создании PHP-классов напоминать про строчные имена файлов для Bitrix autoload.
- Требовать проверку существующей структуры модуля или компонента перед проектированием новых контрактов.
- Требовать batch preload или static cache для списков, чтобы избегать N+1 запросов.
- Требовать инвалидацию кэша, если код изменения данных обновляет кэшируемые данные.
- Требовать сохранять существующие контракты компонентов и AJAX, если задача явно не меняет их.

## planning

- Запрашивать артефакт плана, когда пользователь просит план, архитектуру или декомпозицию задачи.
- Требовать сохранение проектных планов в `.claude/plans/YYYY-MM-DD_<name>.md`, если не указан другой путь.
- Включать зависимости, риски, шаги проверки и критерии завершения.
- Избегать деталей реализации, которые нельзя подтвердить из доступного контекста.

## review

- Начинать с находок, отсортированных по серьёзности.
- Указывать файл и строку, когда это возможно.
- Фокусироваться на багах, регрессиях, безопасности, производительности и недостающих тестах.
- Держать summary вторичным по отношению к находкам.
- Ясно писать, если проблем не найдено, и указывать остаточные риски.

## docs

- Определять целевого читателя и цель документа.
- Предпочитать краткую структуру большим повествовательным блокам.
- Включать точные команды, пути, payload API и формы ответов, когда это релевантно.
- Разделять текущее поведение, желаемое поведение, использование, ограничения и открытые вопросы.

## frontend

- Требовать согласованность с существующей дизайн-системой и библиотекой компонентов.
- Требовать responsive-проверки для mobile и desktop.
- Требовать, чтобы текст не перекрывал элементы и не вылезал из контейнеров.
- Требовать реальные usable-экраны вместо marketing-style заглушек, если landing page не запрошен явно.
- Требовать иконки из существующей библиотеки, когда она доступна.

# Конструктор промптов

Создавай готовый к использованию промпт из короткого описания задачи, большого вставленного текста или локального файла.

## Входные данные

Принимай любой из форматов:
- Прямой текст задачи в `$ARGUMENTS`.
- Локальный путь к файлу с задачей, требованиями, заметками или черновиком промпта.
- Опциональные флаги:
  - `--rules <names>`: наборы правил из `references/rules.md`, перечисленные через запятую.
  - `--agent <target>`: целевой исполнитель, например `codex`, `claude`, `chatgpt`, `dev-agent`, `reviewer`.

Если входных данных нет или они слишком неоднозначны для полезного промпта, задай один короткий уточняющий вопрос.

## Процесс

### Шаг 1: Загрузи наборы правил

Прочитай `references/rules.md`.

Используй:
- правила `core` по умолчанию;
- дополнительные наборы правил, указанные в `--rules`;
- проектные или доменные наборы правил, если задача явно относится к области, описанной в `references/rules.md`.

Если запрошенный набор правил не найден, продолжай с `core` и упомяни отсутствующий набор в финальной заметке.

### Шаг 2: Собери материал задачи

Если `$ARGUMENTS` содержит существующий читаемый путь к файлу:
1. Прочитай файл.
2. Считай содержимое файла исходной задачей.
3. Используй окружающий контекст проекта только если он прямо релевантен и его можно быстро проверить.

Если `$ARGUMENTS` содержит прямой текст:
1. Сохрани все явно указанные требования.
2. Извлеки ожидаемые результаты, ограничения, целевую аудиторию, файлы, команды и критерии успеха.
3. Не придумывай бизнес-требования, которых нет в исходных данных.

### Шаг 3: Нормализуй запрос

Определи:
- Цель: что должно быть достигнуто.
- Контекст: репозиторий, система, домен или фон артефакта.
- Входные данные: текст, файлы, URL, данные, логи, примеры.
- Ограничения: технологии, стиль, язык, совместимость, запрещённые изменения.
- Формат результата: патч кода, анализ, план, документ, чеклист, JSON и т.д.
- Проверку: тесты, команды, ручные проверки, скриншоты, критерии ревью.
- Открытые вопросы: только то, что реально влияет на выполнение.

### Шаг 4: Сформируй промпт

Возвращай промпт на русском языке, если пользователь не попросил другой язык.

Используй структуру:

```markdown
# Роль
<кем должен быть целевой агент>

# Задача
<ясная цель>

# Контекст
<только релевантный контекст>

# Входные данные
<переданный текст, файлы или данные>

# Требования
<обязательные требования>

# Ограничения
<технологии, рамки, безопасность, стиль, совместимость>

# План работы
<пошаговое руководство по выполнению>

# Проверка результата
<как проверить завершение>

# Формат ответа
<точная форма ожидаемого ответа>

# Вопросы, если данных недостаточно
<когда спрашивать, а не додумывать>
```

Опускай разделы, которые не добавляют пользы, но сохраняй результат достаточно полным, чтобы другой агент мог выполнить задачу без чтения исходного диалога.

## Критерии качества

- Промпт должен быть исполнимым, а не мотивационным.
- Сохраняй намерение пользователя и точные формулировки там, где они важны.
- Делай скрытые предположения явными.
- Предпочитай измеримые критерии приёмки.
- Отделяй обязательные требования от желательных улучшений.
- Указывай пути к репозиторию и файлам ровно так, как они даны.
- Если задача связана с кодом, требуй сначала изучить существующие паттерны.
- Если задача связана с созданием артефактов, укажи куда их сохранить и как проверить.
- Если задача рискованная или недостаточно определённая, инструктируй целевого агента сначала спросить, а не действовать наугад.

## Финальный ответ

Верни:
1. Сгенерированный промпт в fenced-блоке `markdown`.
2. Короткую заметку:
   - применённые наборы правил;
   - отсутствующие или нерешённые входные данные, если есть;
   - путь к исходному файлу, если использовался файл.

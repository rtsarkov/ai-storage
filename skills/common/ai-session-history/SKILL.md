# История AI-сессий

Сохраняй короткую, безопасную для передачи историю работы агента в проекте.

## Назначение

Skill ведёт локальный каталог `.aisessions/`:

```text
.aisessions/
├── sessions.json
└── sessions/
    └── <session_hash>.md
```

`sessions.json` — индекс с хэшем сессии, кратким описанием, статусом, тегами и затронутыми областями.
`sessions/<session_hash>.md` — краткая карточка: чем агент занимался в этой сессии.

## Входные данные

Принимай:
- `restore <session_hash>` — восстановить контекст ранее записанной сессии;
- краткое описание сессии из `$ARGUMENTS`;
- опционально `--hash <session_hash>`, если пользователь или среда явно дали hash текущей сессии;
- опционально `--summary <текст>`, если пользователь передал готовое описание для markdown-файла;
- опционально `--status <status>`: `done`, `partial`, `blocked`, `needs-check`;
- опционально повторяемые `--tag <tag>` для поиска по темам;
- опционально повторяемые `--touched <путь или область>` для индекса;
- опционально повторяемые `--decision`, `--check`, `--next` для обогащения карточки.

Если описание не передано, сформулируй его сам по текущему диалогу в 1 короткую фразу.

## Восстановление контекста

Если пользователь просит восстановить контекст из сессии или вызывает `restore <session_hash>`:

1. Запусти:

```bash
python3 <skill-dir>/scripts/record_session.py --restore "<session_hash>"
```

2. Прочитай JSON-ответ скрипта:
   - `entry` — запись из `.aisessions/sessions.json`;
   - `summary` — содержимое markdown-карточки;
   - `summary_file` — путь к карточке.
3. Верни пользователю короткое восстановление контекста:
   - описание сессии;
   - ключевые решения;
   - затронутые области;
   - проверки;
   - что продолжить.
4. Если пользователь сказал "продолжи", используй восстановленный контекст как стартовую точку и продолжай задачу.
5. Если записи нет, сообщи, что hash не найден, и предложи проверить `.aisessions/sessions.json`.

Не изменяй файлы при восстановлении контекста.

## Определение session_hash

Используй первый доступный вариант:

1. Явный `--hash <session_hash>`.
2. Идентификатор текущей сессии/треда, если он явно доступен в контексте агента или runtime-среде.
3. Если hash недоступен, сгенерируй новый через скрипт без `--hash`; скрипт создаст короткий SHA-256 hash из времени, cwd и описания.

Не выдумывай конкретный hash, если он не виден. Если нужно объяснить пользователю источник, скажи: "hash сгенерирован локально, потому что runtime не отдал явный id сессии".

## Что писать в markdown

Пиши кратко, но достаточно для восстановления контекста другой AI-сессией. Карточка должна отвечать на вопрос: "что уже было решено и что нельзя забыть при продолжении?"

Разрешено указывать:
- какие модули, пакеты или файлы были затронуты;
- словесно, что правилось или анализировалось;
- итоговый статус: сделано, частично сделано, заблокировано, нужна проверка.

Не включай:
- фрагменты кода;
- секреты, токены, cookies, ключи;
- длинные трассировки, выводы команд и приватные данные;
- подробные цепочки рассуждений агента.

Рекомендуемый формат markdown:

```markdown
# Сессия <session_hash>

## Описание
<1-2 предложения>

## Что делали
- <краткий пункт>

## Ключевые решения
- <решение, договорённость или выбранный подход>

## Затронутые области
- <модуль/пакет/файл>

## Проверки
- <что проверено или "Не выполнялись">

## Что продолжить
- <следующий шаг, риск или "Не требуется">

## Статус
<сделано/частично/заблокировано>
```

## Процесс

Используй этот процесс только для записи или обновления истории. Для восстановления см. раздел `Восстановление контекста`.

1. Сформируй `description`: одна фраза, зачем была сессия.
2. Сформируй `summary`: короткий markdown без кода и лишних деталей, но с разделами `Ключевые решения`, `Проверки`, `Что продолжить`.
3. Извлеки для индекса:
   - `status`;
   - `tags`;
   - `touched_areas`;
   - `key_decisions`;
   - `checks`;
   - `next_steps`.
4. Запусти скрипт:

```bash
python3 <skill-dir>/scripts/record_session.py \
  --description "<description>" \
  --summary "<summary>" \
  --status "<status>" \
  --tag "<tag>" \
  --touched "<модуль/файл/область>"
```

Если есть явный hash:

```bash
python3 <skill-dir>/scripts/record_session.py \
  --hash "<session_hash>" \
  --description "<description>" \
  --summary "<summary>" \
  --status "<status>"
```

`<skill-dir>` — каталог активной копии skill, например `.codex/skills/ai-session-history` или `.claude/skills/ai-session-history`.

Если `python3` недоступен, попробуй `python`.

5. Прочитай результат скрипта.
6. Сообщи пользователю:
   - создана новая запись или запись уже существовала;
   - путь к `.aisessions/sessions.json`;
   - путь к markdown-файлу сессии.

## Повторная запись

Если запись с таким `session_hash` уже есть:
- не создавай второй элемент в `sessions.json`;
- обнови `updated_at` и markdown-файл только если передан новый `summary`;
- обнови `status`, `tags`, `touched_areas`, `key_decisions`, `checks`, `next_steps`, если они переданы;
- обязательно сообщи пользователю, что такая запись уже существовала.

## Проверка

После записи проверь:
- `.aisessions/sessions.json` существует и содержит запись;
- `.aisessions/sessions/<session_hash>.md` существует;
- в `sessions.json` заполнены `description`, `status`, `summary_file`, а при наличии данных — `tags` и `touched_areas`;
- markdown не содержит кода, секретов и длинных технических выводов.

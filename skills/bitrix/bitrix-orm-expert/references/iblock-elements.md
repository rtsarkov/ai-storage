# Элементы инфоблоков D7 ORM

## Когда читать

Читай этот файл при работе с `Bitrix\Iblock\Elements\Element*Table`, `fetchObject()`, `fetchCollection()`, свойствами инфоблоков, `ElementCommonContentTable`, `PROPERTY.FILE/ITEM/ELEMENT/SECTION`, а также при замене старого `CIBlockElement` на D7 ORM.

## Источники

- Локальный модуль `bitrix/modules/iblock/lib/` и сгенерированная Entity конкретного инфоблока являются источником истины.
- Практическая статья: <https://mrcappuccino.ru/blog/post/iblock-elements-bitrix-d7>. Она протестирована автором на Bitrix 20.0.1196, поэтому используй ее как ориентир, но сверяй с локальным ядром и схемой.

## Entity-класс инфоблока

- Для работы через D7 у инфоблока должен быть задан API-код.
- Класс имеет форму `\Bitrix\Iblock\Elements\Element{ApiCode}Table`.
- Имя класса можно получить через `\Bitrix\Iblock\Iblock::wakeUp($iblockId)->getEntityDataClass()`.
- Перед выборкой подключи модуль `iblock`.
- Для динамических/неочевидных полей проверяй `Table::getEntity()->getFields()` после инициализации, а не только ожидаемый код свойства.

## Выборка массивом

- `getList()->fetch()/fetchAll()` возвращает строки с полями и property-алиасами.
- В `select` код свойства указывается без префикса `PROPERTY_`.
- Для читаемых ключей задавай алиасы: `['ARTICLE_' => 'ARTICLE']`.
- Массивный режим удобен для плоских read-only отчетов, но ключи свойств могут быть шумными и зависят от алиасов.

## Выборка объектами

- `fetchObject()` используй для одного элемента с object API.
- `fetchCollection()` используй для набора элементов и batch-fill.
- Поля элемента читаются camelCase-getter'ами: `getId()`, `getName()`, `getDetailPicture()`.
- Универсальный `get('FIELD')` допустим для динамического поля, но в прикладном коде со стабильной схемой предпочитай сгенерированный getter.

## Разделы

- `IBLOCK_SECTION` дает основной раздел элемента.
- `SECTIONS` дает коллекцию всех разделов: `getSections()->getAll()`.
- Для списков не делай lazy-загрузку разделов внутри цикла; выбирай или `fill()` нужные relation-поля пачкой.

## Свойства

Главное правило: сначала узнай тип и кратность свойства, затем выбирай getter. Не пытайся угадать форму через `method_exists()`.

### Одиночные свойства

- Текст, число, строка, одиночный список и одиночный файл возвращают объект значения.
- Значение читается через `get<Property>()->getValue()`.
- Описание, если включено, читается через `getDescription()`.
- Примеры формы:

```php
$article = $element->getArticle()->getValue();
$lat = $element->getLat()->getValue();
$enumId = $element->getNewproduct()->getValue();
```

### Множественные свойства

- Множественное свойство возвращает Collection.
- Значения читаются через `get<Property>()->getAll()` и `getValue()` у каждого объекта значения.
- Такая форма помогает избежать размножения строк, характерного для JOIN по множественным значениям в плоской выборке.

```php
$values = [];

foreach ($element->getImages()->getAll() as $propertyValue) {
    $values[] = $propertyValue->getValue();
}
```

### Дополнительные поля свойств

- `PROPERTY.ITEM` подключает значение списочного свойства; дальше доступны `getItem()->getId()`, `getXmlId()`, `getValue()`.
- `PROPERTY.FILE` подключает файл; дальше доступен `getFile()` у значения свойства.
- `PROPERTY.ELEMENT` подключает привязанный элемент.
- `PROPERTY.SECTION` подключает привязанный раздел.
- Для множественной привязки к элементам/разделам обходи коллекцию свойства и вызывай `getElement()`/`getSection()` у каждого значения.

## Select и fill

- В `select` указывай только нужные поля и property-пути.
- Если сначала выбираешь базовые поля, а затем работаешь с объектной коллекцией, дозагружай свойства через `$collection->fill([...])`.
- Для множественных файлов с данными файла выбирай или дозагружай `PROPERTY.FILE`, если нужны не только ID файла.
- Следи за кардинальностью: одиночные property-join могут размножать строки в массивном режиме; object collection и fill обычно безопаснее для множественных свойств.

## Кеш

- ORM-запросы поддерживают `cache` с `ttl`.
- Используй кеш только после анализа изменяемости инфоблока и инвалидации.
- Не включай кеш для данных, которые сразу должны отражать изменения после сохранения, без понятной стратегии сброса.

## Антипаттерны

- `method_exists($property, 'getValueList')` как основной способ понять форму свойства.
- Один helper для одиночных и множественных свойств без знания схемы.
- Lazy `get<Property>()`/`fill()` внутри цикла списка.
- `select => ['*']` или выбор всех тяжелых свойств без необходимости.
- Использование статьи или внешнего примера без сверки с локальным `bitrix/modules/iblock/lib/` и реальной Entity.

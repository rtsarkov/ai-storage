# Bitrix Framework Docs

Этот навык помогает агенту использовать официальную документацию Bitrix Framework как практический источник решений: архитектура, модули, контроллеры, ORM, база данных, кеширование, события, компоненты и JS-расширения.

Используй навык, когда нужно спроектировать или проверить Bitrix D7-код, выбрать API, объяснить поведение фреймворка, исправить ошибку в контроллере/ORM/кеше/модуле или подготовить рекомендацию по официальным правилам Bitrix.

Результат: конкретное решение или ревью с привязкой к применимому API, версиям/ограничениям, рискам совместимости и проверкам перед внедрением.

## Step 0: Project Context

1. Если есть `.ai-factory/config.yaml`, прочитай `language.ui`, `language.artifacts`, `language.technical_terms`.
2. Если есть `.ai-factory/skill-context/bitrix-framework-docs/SKILL.md`, прочитай его как проектный override.
3. В проекте с локальным ядром Bitrix сначала проверь реальный код ядра и локальные паттерны проекта. Официальная документация задает направление, но локальная версия и проектные обертки имеют приоритет.

## Workflow

1. Определи область задачи: архитектура, модуль, компонент, контроллер, ORM, SQL, кеширование, событие, JS-расширение или безопасность.
2. Сверь задачу с локальным кодом: версия PHP/Bitrix, namespace модуля, доступные классы, существующий стиль автозагрузки, обработчики, кеши и компоненты.
3. Выбери современный D7-путь, если проект не требует legacy-совместимости: `Loader`, `DataManager`, `Query`, `Engine\Controller`, `EventManager`, `Application::getConnection()`, `UI\Extension`.
4. Проверь границы: права доступа, CSRF/auth/scope filters, SQL escaping, кеш-инвалидацию, N+1 и декартово произведение в ORM.
5. Сформируй решение в стиле проекта: минимальный diff, существующие сервисы/репозитории, Bitrix-автозагрузка, короткие имена файлов классов при необходимости.
6. Перед финалом выполни доступные проверки: `php -l`, точечный unit/ручной сценарий, SQL-план/лог запросов, очистка или инвалидация кеша, проверка ajax/response shape.

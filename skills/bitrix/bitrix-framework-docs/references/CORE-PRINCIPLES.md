# Core Principles

## Приоритеты выбора решения

1. Локальный код проекта и версия ядра важнее общего рецепта. Если официальный пример использует API, которого нет в текущем ядре, найди эквивалент в локальном `bitrix/modules/*`.
2. Новый код размещай в `/local/`: модули в `/local/modules`, компоненты в `/local/components`, JS-расширения в `/local/js`.
3. Используй модульный монолит: бизнес-логика в сервисах/репозиториях модуля, отображение в компонентах/шаблонах, HTTP-логика в контроллерах.
4. Не смешивай физические страницы, SQL, HTML и бизнес-правила в одном файле, если задача не является legacy-поддержкой.

## Архитектура и модули

- Модуль отвечает за отдельную область: API, бизнес-логику, компоненты, админку, события, настройки.
- Идентификатор модуля вида `partner.module` отображается в namespace `Partner\Module`.
- `include.php` выполняется после `Loader::includeModule('partner.module')`; регистрируй там классы, функции и совместимые autoload-настройки.
- Для контроллеров модуля настрой `controllers.defaultNamespace` в `.settings.php`.
- Установочные действия модуля должны быть обратимыми: регистрация модуля, установка таблиц/файлов/обработчиков, удаление при uninstall.

## Контроллеры и HTTP

- Для новых HTTP endpoint предпочитай `Bitrix\Main\Engine\Controller`.
- Действия оформляются методами `*Action`; идентификатор AJAX-действия строится из namespace, класса и метода.
- `configureActions()` настраивает prefilters/postfilters для конкретных действий.
- По умолчанию контроллеры используют фильтры HTTP method, authentication и CSRF. Не отключай их глобально без явной причины.
- Для JSON по умолчанию можно возвращать массив, но явный `Response\Json` нужен, когда требуется ответ без стандартных `status/errors`.
- Для HTML/компонентов используй response-объекты: `Response\Component`, `renderComponent`, `renderComponentAjax`, `renderView`, если они доступны в версии ядра.

## ORM и база данных

- Для таблиц используй `DataManager`: `getTableName()`, `getMap()`, `getUfId()` при пользовательских полях.
- Получай актуальную карту через `Table::getEntity()->getFields()`, особенно если поля добавляются runtime-логикой.
- Для выборок используй `getList()` или `Table::query()`; для сложных условий предпочитай fluent query и `ConditionTree`.
- В отношениях 1:N и N:M не совмещай бездумно `LIMIT` и несколько множественных связей: SQL ограничит строки, а не корневые объекты.
- При нескольких множественных связях контролируй декартово произведение. Используй разложение запроса, batch preload или отдельные relation-запросы.
- Сырой SQL допустим для миграций, batch-операций и оптимизации, но значения и identifiers должны идти через `SqlHelper` или `SqlExpression`.

## Кеширование

- Неуправляемый кеш подходит для дорогих данных с приемлемым TTL и ручной инвалидацией.
- Управляемый кеш подходит для данных, которые должны сбрасываться при изменении таблицы через ORM.
- Компонентный кеш должен учитывать параметры, site id, шаблон, пользователя/права, если результат зависит от них.
- При изменении данных укажи, какой кеш нужно сбросить: ORM table cache, managed cache tag, component cache или статический кеш запроса.
- Для часто меняющихся данных кеш может ухудшить корректность; сначала оцени TTL и инвалидацию.

## События

- Постоянные обработчики регистрируй при установке модуля через `EventManager::registerEventHandler`.
- При удалении модуля удаляй обработчики через `unRegisterEventHandler`.
- `addEventHandler` используй только для осознанной динамической регистрации; она хуже для анализа и отладки.
- Legacy-события `OnBefore*` могут передавать аргументы без объекта `Event`; обрабатывай их как совместимый слой.

## JS-расширения

- JS/CSS организуй как Bitrix UI Extension: `src`, `dist`, `bundle.config.js`, `config.php`.
- Загружай расширения через `\Bitrix\Main\UI\Extension::load('module.extension')`.
- Зависимости импортируй в JS, чтобы сборщик добавлял их в `config.php`, если используется Bitrix CLI.
- Для редких сценариев применяй `Runtime.loadExtension()`, чтобы не грузить код на первом рендере.

## Failure Modes

- Отключен CSRF/auth filter в endpoint, который меняет данные.
- SQL собирается конкатенацией пользовательских значений.
- `setLimit()` применяется к запросу с множественной связью и дает неполную коллекцию.
- Компонент кеширует результат без учета прав доступа или пользователя.
- Обработчик события регистрируется динамически в рантайме и дублируется при каждом хите.
- Новый код добавлен в `/bitrix/`, из-за чего будет потерян при обновлении.
